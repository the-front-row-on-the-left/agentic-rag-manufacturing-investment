# Git 커밋 및 머지 전략

이 문서는 본 프로젝트에서 사용하는 Git 커밋 규칙과 머지 전략을 정의한다.  
목적은 변경 이력을 명확하게 유지하고, 협업 과정에서 충돌과 히스토리 오염을 줄이는 데 있다.

---

## 1. 기본 원칙

- 모든 변경 사항은 브랜치에서 작업한다.
- `main` 브랜치에는 직접 커밋하거나 직접 머지하지 않는다.
- `main` 브랜치 반영은 **Pull Request(PR)** 를 통해서만 수행한다.
- PR을 올리기 전에는 항상 최신 `main` 기준으로 **rebase** 한다.

---

## 2. 커밋 타입 규칙

커밋 메시지는 아래 타입 중 하나로 시작한다.

### 허용 타입

- `feature`: 새로운 기능 추가
- `fix`: 버그 수정
- `chore`: 빌드, 설정, 의존성, 잡무성 변경
- `docs`: 문서 작성 및 수정
- `style`: 로직 변경 없는 스타일 수정
- `delete`: 파일 또는 기능 제거
- `refactor`: 동작 변화 없는 구조 개선

---

## 3. 커밋 메시지 형식

권장 형식은 아래와 같다.

```text
type: 변경 내용 요약
```

예시:

```text
feature: 제조업 AI 스타트업 탐색 에이전트 추가
fix: 시장 분석 에이전트의 JSON 파싱 오류 수정
docs: Git 전략 문서 추가
style: Ruff 포맷 적용
delete: 사용하지 않는 예제 스크립트 제거
refactor: GraphState 분리 로직 단순화
chore: pre-commit 설정 추가
```

---

## 4. 커밋 메시지 작성 규칙

- 한 커밋은 하나의 목적만 가지도록 작성한다.
- 제목만 봐도 변경 목적을 이해할 수 있도록 작성한다.
- 너무 많은 변경을 하나의 커밋에 몰지 않는다.
- 의미 없는 메시지(`update`, `test`, `fix bug`)는 지양한다.
- 문서 수정, 리팩토링, 기능 추가를 한 커밋에 섞지 않는다.

---

## 5. 브랜치 전략

권장 브랜치명 예시는 다음과 같다.

```text
feature/startup-search-agent
fix/json-parse-error
docs/git-strategy
refactor/scoring-agent
chore/pre-commit-config
```
---

## 6. 머지 전략

### main 브랜치 보호 규칙

- `main`에는 직접 push 하지 않는다.
- `main`에는 **PR을 통해서만 머지 가능**하도록 설정한다.
- 가능하다면 branch protection rule을 적용한다.

### PR 전 준비 사항

PR을 생성하기 전 반드시 다음을 수행한다.

1. 최신 `main`을 fetch 한다.
2. 현재 작업 브랜치를 `main` 기준으로 rebase 한다.
3. 충돌을 해결한다.
4. 테스트 및 pre-commit 검사를 통과시킨다.

예시:

```bash
git fetch origin
git checkout feature/my-branch
git rebase origin/main
```

---

## 7. PR 머지 원칙

- PR은 최신 `main` 기준으로 rebase 된 상태여야 한다.
- 리뷰 후 머지한다.
- 가능하면 작은 단위의 PR로 나누어 올린다.
- PR 설명에는 다음 내용을 포함하는 것을 권장한다.
  - 변경 목적
  - 주요 변경 사항
  - 테스트 여부
  - 리뷰 포인트

---

## 8. 권장 머지 방식

본 프로젝트에서는 히스토리를 깔끔하게 유지하기 위해 다음을 권장한다.

- 작업 브랜치 → `main` 반영 전 **rebase**
- PR 머지 시에도 가능하면 **불필요한 merge commit을 최소화**

팀 설정에 따라 최종 머지 버튼은 아래 중 하나를 선택할 수 있다.

- Rebase and merge
- Squash and merge

둘 중 무엇을 사용할지는 팀의 히스토리 관리 정책에 따라 정하되,  
현재 원칙은 **최신 main 기준으로 rebase 후 PR 머지**이다.

---

## 9. 충돌 해결 원칙

- 충돌은 가능한 한 작업 브랜치에서 미리 해결한다.
- 충돌 해결 후에는 주요 기능이 정상 동작하는지 다시 확인한다.
- 충돌 해결만을 위한 별도 의미 없는 커밋을 남기기보다, 정리된 상태로 rebase 완료 후 PR을 올리는 것을 권장한다.

---

## 10. 리뷰 및 품질 체크 권장 사항

PR 머지 전 아래를 확인하는 것을 권장한다.

- pre-commit 통과
- 테스트 통과
- 불필요한 로그/주석 제거
- 임시 파일 제거
- 문서 변경 필요 여부 확인

---

## 11. 추가 권장 사항

### 1) 기능 단위 커밋
기능 추가, 버그 수정, 문서 수정은 가능한 한 분리된 커밋으로 관리한다.

### 2) PR 크기 제한
너무 큰 PR은 리뷰 품질을 떨어뜨리므로, 가능한 한 작은 단위로 분리한다.

### 3) 강제 push 주의
rebase 이후에는 `push --force-with-lease`를 사용할 수 있으나,  
공유 브랜치에서는 팀원과 반드시 조율한다.

예시:

```bash
git push --force-with-lease
```

### 4) 작업 전 동기화
작업 시작 전 최신 `main`을 먼저 반영하면 충돌 가능성을 줄일 수 있다.

---

## 12. 운영 원칙 요약

- 커밋 타입은 `feature`, `fix`, `chore`, `docs`, `style`, `delete`, `refactor`만 사용한다.
- 커밋 메시지는 `type: 변경 내용 요약` 형식을 따른다.
- `main` 브랜치에는 직접 머지하지 않는다.
- 모든 변경은 PR을 통해서만 `main`에 반영한다.
- PR 전에는 항상 최신 `main` 기준으로 rebase 한다.
- pre-commit 및 테스트를 통과한 뒤 PR을 머지한다.
