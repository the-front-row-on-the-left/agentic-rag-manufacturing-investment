# Demo Scenarios

## 목적

이 문서는 발표와 시연에서 사용할 입력값, 기대 결과, 확인 포인트, 실패 대응 멘트, 완료 기준, 체크리스트, 발표 분담을 사전에 정리한 운영 문서다.  
목표는 데모 중 오류 발생 시에도 설명 흐름과 역할 분담이 흔들리지 않도록 만드는 것이다.

## 시나리오 1. 비전 검사 스타트업 탐색

| 항목 | 내용 |
| --- | --- |
| 시나리오명 | 비전 검사 스타트업 탐색 |
| 입력값 | `keyword="manufacturing AI inspection startup"` |
| 기대 결과 | 후보 탐색, 기업 프로필 생성, 기술/시장/경쟁 분석, 투자 판단 도출 |
| 확인 포인트 | 후보 수가 생성되는가, `startup_profile` 이 채워지는가, references 가 포함되는가 |
| 실패 시 대응 | 기업 정보가 부족하면 공개 정보 기준의 보수적 요약으로 설명한다 |
| 완료 기준 | 최종 보고서에 대상 기업 개요, 기술 분석, 시장 분석, decision 이 모두 포함됨 |

### 발표 멘트 포인트

- 검색 단계에서 제조업 비전 검사 관련 후보가 어떻게 추려지는지 설명한다.
- 이후 기업 요약과 기술 분석이 연결되는 흐름을 강조한다.

## 시나리오 2. 예지보전 스타트업 평가

| 항목 | 내용 |
| --- | --- |
| 시나리오명 | 예지보전 스타트업 평가 |
| 입력값 | `keyword="predictive maintenance startup"` |
| 기대 결과 | 기술/시장 분석 후 투자 판단 생성 |
| 확인 포인트 | RAG 근거가 반환되는가, 시장/기술 분석이 모두 생성되는가, decision 이 도출되는가 |
| 실패 시 대응 | 검색 결과 편차가 크면 키워드를 `industrial predictive maintenance AI startup` 으로 재조정한다 |
| 완료 기준 | 기술 적용성, 시장 수요, 경쟁 구도까지 포함된 decision 이 생성됨 |

### 발표 멘트 포인트

- RAG 문서가 예지보전의 ROI 와 도입 장벽 설명에 어떻게 기여하는지 보여준다.
- 기술 분석과 시장 분석이 별도 흐름으로 진행된다는 점을 설명한다.

## 시나리오 3. 보류 케이스 확인

| 항목 | 내용 |
| --- | --- |
| 시나리오명 | 보류 케이스 |
| 입력값 | `keyword="industrial AI niche startup"` |
| 기대 결과 | `hold` 또는 `conditional_review` 사례 확인 |
| 확인 포인트 | 투자 판단 로직이 보수적으로 작동하는가, 근거 부족 시 리스크가 명시되는가 |
| 실패 시 대응 | 정보가 부족해도 억지 추천을 하지 않고 보수적 판단 근거를 설명한다 |
| 완료 기준 | 보류 또는 조건부 검토 판정과 그 이유가 명시적으로 출력됨 |

### 발표 멘트 포인트

- 시스템이 항상 추천하지 않는다는 점을 보여준다.
- 근거 부족 자체를 리스크로 반영하는 설계를 강조한다.

## 완료 기준

| 역할 | 완료 기준 |
| --- | --- |
| 1번 | LangGraph 흐름이 정상 연결되고, 후보 반복 평가 및 라우팅이 동작함 |
| 2번 | RAG 인덱스가 정상 생성되고, 기술/시장 검색에 일관된 chunk 와 reference 를 제공함 |
| 3번 | 기술/시장/경쟁 분석과 투자 판단 JSON 이 스키마대로 생성됨 |
| 4번 | 최종 markdown 보고서가 합의된 형식으로 생성되고, 데모 시나리오별 설명 흐름이 준비됨 |

## 체크리스트

### 실행 전

- `.env` 파일이 존재하는가
- `OPENAI_API_KEY` 가 설정되어 있는가
- `TAVILY_API_KEY` 가 설정되어 있는가
- `data/rag_docs/manifest.json` 이 존재하는가
- PDF 원문이 `data/rag_docs/` 아래 존재하는가
- Qdrant 서버가 실행 중인가

### 실행 중

- 입력 키워드가 발표 시나리오와 맞는가
- 네트워크 연결이 정상인가
- API 응답 지연이 심하지 않은가
- 후보가 0건으로 떨어지지 않는가

### 실행 후

- `outputs/final_report_*.md` 가 생성되었는가
- `outputs/final_state_*.json` 이 생성되었는가
- 보고서에 References 섹션이 포함되었는가

### 1번 시작부 검증 포인트

- `candidate_startups` 가 비어 있지 않고 제조업 AI 후보로 읽히는가
- `selected_startup` 이 현재 또는 마지막 평가 대상 포인터로 해석되는가
- `startup_profile` 이 채워지고, 회사 개요와 핵심 제품이 일관되게 요약되는가
- 첫 후보가 `hold` 또는 `conditional_review` 일 때 다음 후보로 이동한 흔적이 `current_index`, `evaluation_history` 에 남는가
- 최종 추천 결과는 `selected_startup` 이 아니라 `recommended_startups` 와 `evaluation_history` 로 읽히는가

## 발표 분담

| 담당 | 발표 구간 |
| --- | --- |
| 1번 | 전체 아키텍처, 그래프 흐름, 상태 전달 방식 설명 |
| 2번 | RAG 문서 구성, 인덱싱, retrieval 근거 설명 |
| 3번 | 기술/시장/경쟁 분석과 투자 판단 로직 설명 |
| 4번 | 최종 보고서, 데모 시나리오 진행, 결과 해석 |

## 장애 대응

### API 지연

- 대응 멘트: `현재 외부 API 응답이 지연되고 있어, 사전에 생성된 결과와 흐름 중심으로 설명드리겠습니다.`

### 검색 실패

- 대응 멘트: `검색 키워드 편차가 있어 보다 구체적인 산업 키워드로 재시도하겠습니다.`
- 대체 입력 예시:
  - `manufacturing AI inspection startup`
  - `industrial predictive maintenance AI startup`

### 문서 근거 부족

- 대응 멘트: `현재 코퍼스는 제조업 일반 보고서 중심이므로, 특정 스타트업 직접 정보는 공개 웹 근거와 함께 보수적으로 해석하겠습니다.`

### 기업 정보 부족

- 대응 멘트: `공개된 회사 정보가 제한적이어서, 확인 가능한 제품/시장 정보 중심으로 최소 안전 범위만 요약하겠습니다.`

## 데모 진행 체크리스트

- 실행 명령어를 미리 준비했는가
- 시나리오별 입력 키워드를 복사해 두었는가
- 발표자별 말할 순서를 정했는가
- 오류 발생 시 대체 멘트를 합의했는가
- 최종 보고서 예시 파일을 백업으로 준비했는가
